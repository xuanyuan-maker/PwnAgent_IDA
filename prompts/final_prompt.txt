格式要求：
- `Vulnerability Type`：只写一句简短结论，不要表格，不要代码块。
- `Primary Suspicious Site`：先写你认为最异常的一个调用点或判断点，必须包含真实函数名，并尽量引用关键语句/API/常量；先描述“哪里最反常”，再谈漏洞归类。
- `Vulnerability Type`：只写一句简短结论。若暂时更像“异常调用/参数错位/危险状态破坏”而不是经典漏洞名，可以直接如实描述，不要强行套常见类别。
- `Vulnerable Functions`：使用 bullet 列出函数名；若能确认漏洞函数，至少列出 1 个。
- `Root Cause`：只写根因，不要写泛泛风险描述。
- `Trigger Condition`：写清楚触发路径或条件；若是程序启动即触发，明确写出。
- `Key Evidence`：使用 bullet，每条都引用具体证据，优先写“函数名 + 关键语句/API/常量”，不要只写泛化结论。
- `Impact`：只写影响结论，不要表格。
- `False Positive Risk`：只写 `low` / `medium` / `high` 之一。
- `Patch Idea`：写修复思路，不要表格。
- `Minimal Fix`：写最小修复动作；若无需代码级补丁，也要给明确动作。
- `Manual Checks`：使用 bullet；若没有额外复核项，写 `- 无`。
- 不要输出其他标题，不要输出 Markdown 表格，不要输出分隔线。

特别要求：
- 若证据显示程序直接调用 `system("/bin/sh")` 或等价固定 shell 启动逻辑，应明确判定为“直接执行 shell / 后门式任意命令执行风险”。
- 这类场景即使没有用户输入，也不能因为“缺少输入点”而回避结论。
- 先找“最反常的一行/一个调用点”，再决定它属于哪类漏洞；不要先套漏洞模板再反推证据。
- 对菜单堆题，不要默认优先输出 UAF 或 size 校验问题；若 `edit/modify/fill` 路径出现更异常的写入/格式化调用，应优先围绕该调用点展开。
- `Root Cause` 必须明确写出“哪个函数把哪个用户可控长度/指针传给了哪个危险写入点，且缺少了什么检查”。
- `Key Evidence` 至少写 2 条，且每条都必须包含证据包里的真实函数名或 API 名。
